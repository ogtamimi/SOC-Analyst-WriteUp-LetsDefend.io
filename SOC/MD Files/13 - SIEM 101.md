Security information and event management (SIEM) is a security solution that collects and interprets data within the organization and then detects potential threats. ## 2) Log Collection
In computing, a log file is a file that records either events that occur in an operating system or other software runs, or messages between different users of a communication software. Logging is the act of keeping a log. In the simplest case, messages are written to a single log file. definition: wikipedia.org

Our goal at this point is to transfer logs from various places (Hosts, Firewall, Server log, Proxy, etc.) to SIEM. Thus, we can process all data and detect threats at a central point. Logs are generally collected in the following 2 ways:
* Log Agents
* Agentless

### Log Agents
In order to implement this method, a log agent software is required. Agents often have parsing, log rotation, buffering, log integrity, encryption, conversion features. In other words, this agent software can take action on the logs it collects before forwarding them to the target. For example, with the agent software, we can divide a log with "username: LetsDefend; account: Administrator" into 2 parts and forward it as:

* message1 = "username: LetsDefend"
* message2 = "account: Administrator"

| **Category** | **Details** |
|--------------|-------------|
| **Pros of the Method** | - Tested and working application by developers<br>- Many additional features like automatic parsing, encryption, log integrity, etc. |
| **Cons of the Method** | - Additional features increase resource consumption<br>- Requires higher CPU, RAM, and system resources → higher cost |
| **Syslog** | - Popular network protocol for log transfer<br>- Works over UDP or TCP, optionally encrypted with TLS<br>- Supported by Switch, Router, IDS, Firewall, Linux, Mac; Windows can support with additional software<br>- Log format: `Timestamp - Source Device - Facility - Severity - Message Number - Message Text`<br>- Max packet size: UDP = 1024 bytes, TCP = 4096 bytes |
| **Third-Party Agents** | - Most SIEM products have their own agents<br>- Provide more capabilities than plain syslog<br>- Examples:<br> &nbsp;&nbsp;- Splunk: Universal Forwarder<br> &nbsp;&nbsp;- ArcSight: ArcSight Connectors<br>- Easy integration with SIEM and support log parsing |
| **Popular Open Source Agents** | - [Beats](https://www.elastic.co/beats/)<br>- [NXLog](https://nxlog.co/) |

### Agentless
Agentless log sending process is sometimes preferred as there is no installation and update cost. Usually, logs are sent by connecting to the target with SSH or WMI. For this method, the username and password of the log server are required, therefore there is a risk of the password being stolen. Easier to prepare and manage than the agent method. However, it has limited capabilities and credentials are wrapped in the network.

## 3) Log Aggregation and Parsing
The first place where the generated logs are sent is the log aggregator. We can edit the logs coming here before sending them to the destination. For example, if we want to get only status codes from a web server logs, we can filter among the incoming logs and send only the desired parts to the target.

Aggregator EPS

### What is EPS?

The acronym EPS stands for events per second. The EPS formula is Events/Time Period (seconds). For example, if the system receives 1,000 logs in five seconds, the EPS would be 200. As the EPS value increases, so should the aggregator and storage capacity.


### Scaling the Aggregator

More than one aggregator can be added so that the incoming logs do not load the same aggregator each time. And sequential or random selection can be provided.
![alt text](../Assets/lab13/3.3.png)

### Log Aggregator Process

The log coming to the Aggregator is processed and then directed to the target. This process can be parsing, filtering, and enrichment.

![alt text](../Assets/lab13/3.4.png)

Log Modification

In some cases, you need to edit the incoming log. For example, while the date information of most logs you collect comes in the format dd-mm-yyyy, if it comes from a single source as mm-dd-yyyy, you would want to convert that log. Another example, you may need to convert UTC + 2 incoming time information to UTC + 1.


Log Enrichment
Enrichment can be done to increase the efficiency of the collected logs and to save time. Example enrichments:



* Geolocation
* DNS
* Add/Remove


Geolocation
The geolocation of the specified IP address can be found and added to the log. Thus, the person viewing the log saves time. It also allows you to analyze location-based behavior.


DNS
With DNS queries, the IP address of the domain can be found or the IP address can be found by doing reverse DNS.

## 4) Log Storage

In the previous sections, we discussed **logs** and **log aggregators**. The next step in a SIEM architecture is **storing the collected logs efficiently**.

![Log Storage Architecture](../Assets/lab13/4.1.png)

One common mistake in SIEM implementations is focusing only on **storage capacity**. While having large storage space is important, **data access speed** is just as critical.

For example, imagine collecting logs from multiple sources such as **WAF, Firewalls, and Proxy servers**. If searching through these logs takes **15 minutes**, security investigations and analysis become inefficient. During an incident, analysts need to retrieve information quickly to understand what happened and respond appropriately. Therefore, storage systems used in SIEM environments must ensure **fast and reliable data retrieval**.

Traditional storage technologies such as **MySQL** are designed mainly for **adding, updating, and deleting records**. However, SIEM systems have different requirements. Security logs are usually **not modified after they are stored**; instead, they are frequently searched and analyzed.

For this reason, **WORM (Write Once, Read Many)** based storage technologies are more suitable for SIEM systems. These systems allow data to be written once and accessed many times without modification, making them ideal for **log indexing and fast search operations**.

More information about WORM (Write Once, Read Many):  
https://en.wikipedia.org/wiki/Write_once_read_many

## 5) Alerting
We have collected, processed and stored logs up to this point. Now, we need to detect abnormal behavior using the data we have and generate alerts.

![alt text](../Assets/lab13/5.1.png)


Timely occurrence of alerts varies depending on our search speed. For a log created today, we want to create a warning immediately instead of generating an alert after 2 days. Therefore, as we mentioned in our previous article, a suitable storage environment should be created. The alarms we will create for SIEM will usually be suspicious and need to be investigated. This means that the alert must be optimized and not triggered in large numbers (except in exceptional cases). Here are some ways to create an alert:
* By searching stored data
* Creating alarms while taking logs

| Method | Description |
|------|-------------|
| **Whitelist** | Unlike a blacklist, a whitelist is used to define **allowed or trusted entities**. For example, a list of IP addresses that are expected to communicate normally can be maintained. If communication occurs with an IP address outside this list, an **alert can be generated**. This approach can be highly effective for detecting unauthorized activity, but it is **difficult to manage** because the list must be **continuously updated**. |
| **Long Tail Log Analysis** | This method assumes that **frequently occurring behaviors are normal**. For example, if the log *"Event ID 4624 – An account was successfully logged on"* appears regularly on a device, it is considered normal behavior. With this approach, analysts focus more on **rare or unusual events**, since these may indicate suspicious or malicious activity. |
